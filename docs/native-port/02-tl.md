# Vianium.Tl (`vianium-mtproto\src\tl\`) — Type Language Serialization

**Status:** planned (Phase 1) · **Layer baseline:** 214 · **Last reviewed:** 2026-04-27 · **Sibling repo:** `vianium-mtproto` (`src\tl\`)

> Required prior reading:
> - [architecture-principles.md](architecture-principles.md) — toolchain, DDD+hex, WinMD ABI rules.
> - [00-overview.md](00-overview.md) — context map.
> - [01-mtproto.md](01-mtproto.md) — which consumes Tl as the `i_tl_codec` port.

The `src\tl\` subcomponent of the `vianium-mtproto` sibling implements the TL ("Type Language") serializer/deserializer that MTProto uses for all of its messages. TL is a custom binary format by Telegram with strong typing and a schema versioned by **layer**. Each TL constructor has a `constructor_id` (uint32) that prefixes the encrypted body. Vianigram consumes the WinMD; this doc describes the API and the invariants.

**Reference spec:** https://core.telegram.org/mtproto/TL. Canonical schema `tl/scheme-layer-214.tl` (committed with the Vianigram repo). Layer 214 fixes the baseline; layer bumps documented in section 11.

---

## 1. Bounded context

### Ubiquitous language

| Term | Definition |
|---------|------------|
| **schema** | The complete set of TL definitions for a given layer. Text in `.tl` format. |
| **constructor** | A concrete definition of a type. Has a constructor_id (uint32), a name, parameters, and a return type. |
| **constructor_id** | The normalized CRC32 hash of the constructor's textual definition. Identifies it uniquely. |
| **layer** | A schema version. Layer 214 = the active one when Vianigram starts (matches PivoraTelegram). |
| **boxed type** | A type whose wire encoding includes the `constructor_id` before the body. The default when there is polymorphism. |
| **bare type** | A type whose wire encoding does NOT include a `constructor_id` (because the context resolves it uniquely). A subset of constructors. |
| **vector** | A homogeneous container. Wire: `0x1cb5c415` + count uint32 + items. |
| **flags** | A uint32 bitfield indicating the presence of optional fields. Syntax `flags.N?Type`. |
| **codegen** | A C# tool that parses `.tl` and emits C++ structs + a serializer. |

### Aggregate root

`TlSchema` (in `domain/aggregates/tl_schema.h`):
- Identified by `layer_number`.
- Owns: `factory_table` (a map of `constructor_id` to a factory function), `name_table` (debug-only, a map to the textual name).
- Invariants:
  - Each `constructor_id` appears only once.
  - The layer is not modified post-load.

### Domain events

- `SchemaLoaded{ layer, constructor_count }`.
- `UnknownConstructor{ constructor_id, context }` — telemetry for unregistered received constructors (useful for detecting layer bumps on the server).

## 2. TL wire format

### Primitive types

| TL type | Wire | Size |
|---------|------|--------|
| `int` | int32 LE | 4 bytes |
| `long` | int64 LE | 8 bytes |
| `int128` | byte[16] | 16 bytes |
| `int256` | byte[32] | 32 bytes |
| `double` | IEEE 754 LE | 8 bytes |
| `string` | length-prefixed UTF-8, padded to a multiple of 4 | variable |
| `bytes` | the same encoding as `string` | variable |
| `Bool` | constructor `boolFalse#bc799737` or `boolTrue#997275b5` | 4 bytes (boxed) |
| `vector<T>` | `0x1cb5c415` + count uint32 + items | variable |

### String / bytes encoding

```
if length < 254:
    [length: 1 byte]  [data: length bytes]  [padding: pad to 4-byte boundary]
else:
    [0xFE: 1 byte]  [length: 3 bytes LE]  [data: length bytes]  [padding: pad to 4-byte boundary]
```

Padding is mandatory. Empty strings = 4 bytes total: `[0x00, 0x00, 0x00, 0x00]`.

### Vector

```
[0x1cb5c415: uint32 LE]  [count: uint32 LE]  [item_0]  [item_1]  ...  [item_{count-1}]
```

`0x1cb5c415` is the canonical constructor_id of `vector`. When a constructor specifies a vector with a concrete item type, the wire does NOT repeat the item constructor_ids if the items are **bare**.

### Flag-based optional fields

Syntax:
```
foo#deadbeef flags:# field1:flags.0?int field2:flags.1?string field3:int = Foo;
```

- `flags:#` declares a `uint32` field that indicates the presence of optionals.
- `field1:flags.0?int` means "include the int if bit 0 of flags is set".
- `field3:int` is always present.

Wire:
```
[constructor_id: uint32]  [flags: uint32]  [optional fields that are set]  [mandatory fields]
```

### Boxed vs bare

**Boxed** (default): the wire includes the `constructor_id`. Required when:
- The parameter is polymorphic (e.g., `Peer` can be `peerUser#9db1bc6d`, `peerChat#bad0e5bb`, `peerChannel#a2a5371e`).
- The parameter is a container with polymorphic items.

**Bare**: the wire does NOT include the `constructor_id`. Only allowed for:
- Private types internal to a container that knows the type by context.
- Some system types like `vector` when the items are primitives.

In practice, layer 214 uses boxed almost exclusively for public types. Bare is seen in internal containers (`%(true)`, etc.).

## 3. Schema codegen

### Tool

`tools/tl-codegen/` is a .NET 4.6.1 console utility app (managed, NOT WP8.1) that is run at build-time:

```

├── TlCodegen/
│   ├── Program.cs
│   ├── Lexer.cs                    ← tokenize .tl
│   ├── Parser.cs                   ← AST of definitions
│   ├── ConstructorIdHasher.cs      ← computes normalized CRC32
│   ├── CppEmitter.cs               ← emits .h + .cpp
│   └── tl/
│       └── scheme-layer-214.tl     ← input
```

Invocation (from MSBuild as `<Exec>`):
```
TlCodegen.exe \
    --input  tools/tl-codegen/tl/scheme-layer-214.tl \
    --layer  214 \
    --header ../../../vianium-mtproto/src/tl/infrastructure/generated/tl_layer_214.h \
    --source ../../../vianium-mtproto/src/tl/infrastructure/generated/tl_layer_214.cpp \
    --namespace vianium::tl::infrastructure::generated::layer_214
```

### Layer flag

Vianigram starts at layer 214 (matches PivoraTelegram). Multi-layer support is possible:

```
src/infrastructure/generated/
├── tl_layer_214.{h,cpp}            ← baseline
└── tl_layer_220.{h,cpp}            ← future, once the upgrade is approved
```

`TlSchema::LoadLayer(int layerNumber)` dispatches to the corresponding table. Only one active layer per `MTProtoChannel`. Upgrade path documented in section 11.

### Generated code shape

For a definition:
```
user#020b1422 flags:# id:long access_hash:flags.0?long first_name:flags.1?string
              last_name:flags.2?string username:flags.3?string phone:flags.4?string = User;
```

The codegen emits (excerpt):
```cpp
// tl_layer_214.h
namespace vianium::tl::infrastructure::generated::layer_214 {

struct TlUser {
    static constexpr uint32_t kConstructorId = 0x020b1422;
    uint32_t flags;
    int64_t id;
    int64_t access_hash;        // valid if flags & (1u << 0)
    std::vector<wchar_t> first_name;  // valid if flags & (1u << 1)
    std::vector<wchar_t> last_name;
    std::vector<wchar_t> username;
    std::vector<wchar_t> phone;

    Result<size_t, TlError> Write(TlWriter& w) const;
    static Result<TlUser, TlError> Read(TlReader& r);
    size_t EstimateSize() const;
};

void RegisterAllLayer214(TlSchema& schema);

} // namespace
```

And the implementation in `tl_layer_214.cpp`:
```cpp
Result<size_t, TlError> TlUser::Write(TlWriter& w) const {
    auto start = w.Position();
    if (!w.WriteUInt32(kConstructorId).IsOk()) return TlError::WriteFailed;
    if (!w.WriteUInt32(flags).IsOk()) return TlError::WriteFailed;
    if (!w.WriteInt64(id).IsOk()) return TlError::WriteFailed;
    if (flags & (1u << 0)) {
        if (!w.WriteInt64(access_hash).IsOk()) return TlError::WriteFailed;
    }
    if (flags & (1u << 1)) {
        if (!w.WriteString(first_name).IsOk()) return TlError::WriteFailed;
    }
    // ... rest ...
    return Result<size_t, TlError>::Ok(w.Position() - start);
}

Result<TlUser, TlError> TlUser::Read(TlReader& r) {
    TlUser obj{};
    auto ctor = r.ReadUInt32();
    if (!ctor.IsOk() || ctor.Value() != kConstructorId) return TlError::WrongConstructor;
    auto flags = r.ReadUInt32();
    if (!flags.IsOk()) return TlError::ReadOverflow;
    obj.flags = flags.Value();
    auto id = r.ReadInt64();
    if (!id.IsOk()) return TlError::ReadOverflow;
    obj.id = id.Value();
    if (obj.flags & (1u << 0)) {
        auto h = r.ReadInt64();
        if (!h.IsOk()) return TlError::ReadOverflow;
        obj.access_hash = h.Value();
    }
    // ... rest ...
    return Result<TlUser, TlError>::Ok(std::move(obj));
}
```

### Constructor ID hashing

Telegram specifies that the `constructor_id` is computed as follows:
1. Take the textual definition without the `#hash` or the `=`/`;`.
2. Normalize whitespace (single spaces).
3. Replace `?` with `?` (literal).
4. Compute the standard CRC32 (polynomial 0xEDB88320).
5. The result = constructor_id as a uint32 LE.

The codegen verifies that the `#hash` declared in the `.tl` matches the computed one, and emits a warning if they differ. PivoraTelegram had some historical mismatch cases due to evolution; Vianigram requires a strict match.

## 4. TlReader / TlWriter

### TlWriter

```cpp
namespace vianium::tl::application {

class TlWriter {
public:
    TlWriter(IAllocator* alloc, size_t initialCapacity = 1024);

    // Primitives
    Result<void, TlError> WriteInt32(int32_t v);
    Result<void, TlError> WriteUInt32(uint32_t v);
    Result<void, TlError> WriteInt64(int64_t v);
    Result<void, TlError> WriteUInt64(uint64_t v);
    Result<void, TlError> WriteInt128(const uint8_t v[16]);
    Result<void, TlError> WriteInt256(const uint8_t v[32]);
    Result<void, TlError> WriteDouble(double v);
    Result<void, TlError> WriteBool(bool v);

    // Strings / bytes (length-prefixed + 4-byte aligned)
    Result<void, TlError> WriteString(const wchar_t* utf16, size_t len);
    Result<void, TlError> WriteString(StringView16 sv);
    Result<void, TlError> WriteBytes(const uint8_t* bytes, size_t len);
    Result<void, TlError> WriteBytes(Span<const uint8_t> bytes);

    // Vector
    Result<void, TlError> WriteVectorBegin(size_t count);     // emits 0x1cb5c415 + count
    template <typename T> Result<void, TlError> WriteVector(const std::vector<T>& items);

    // Object (boxed) — writes constructor_id + body
    template <typename T> Result<void, TlError> WriteObject(const T& obj) {
        return obj.Write(*this);
    }

    // Position management
    size_t Position() const;
    Span<const uint8_t> Bytes() const;
    void Reset();

    // Pre-allocation
    Result<void, TlError> Reserve(size_t bytes);

private:
    IAllocator* m_alloc;
    uint8_t* m_buf;
    size_t m_capacity;
    size_t m_pos;
};

}
```

### TlReader

```cpp
namespace vianium::tl::application {

class TlReader {
public:
    TlReader(Span<const uint8_t> input);

    // Primitives — every read bounds-checks
    Result<int32_t, TlError> ReadInt32();
    Result<uint32_t, TlError> ReadUInt32();
    Result<int64_t, TlError> ReadInt64();
    Result<uint64_t, TlError> ReadUInt64();
    Result<void, TlError> ReadInt128(uint8_t out[16]);
    Result<void, TlError> ReadInt256(uint8_t out[32]);
    Result<double, TlError> ReadDouble();
    Result<bool, TlError> ReadBool();

    Result<StringView16, TlError> ReadString(IAllocator* outAlloc);    // copies the decoded UTF-16 to outAlloc
    Result<Span<const uint8_t>, TlError> ReadBytesView();              // zero-copy view over the input

    Result<size_t, TlError> ReadVectorBegin();    // verifies 0x1cb5c415 + returns count
    template <typename T> Result<std::vector<T>, TlError> ReadVector(IAllocator* outAlloc);

    // Object (boxed): peek constructor_id + dispatch via TlSchema
    template <typename T> Result<T, TlError> ReadObject() {
        return T::Read(*this);
    }

    Result<TlObjectHandle, TlError> ReadAnyObject(TlSchema* schema, IAllocator* outAlloc);

    // Position management
    size_t Position() const;
    size_t Remaining() const;
    Result<void, TlError> Skip(size_t bytes);
    Result<uint32_t, TlError> PeekUInt32();    // without advancing

private:
    Span<const uint8_t> m_input;
    size_t m_pos;
};

}
```

**Bounds checking**: each `Read*` verifies `m_pos + size <= m_input.size`. If it overflows → `TlError::ReadOverflow`.

**Alignment**: `ReadString` and `ReadBytesView` automatically skip the padding at the end.

## 5. Vector encoding details

```
[0x1cb5c415: uint32 LE]  [count: uint32 LE]  [item_0]  [item_1]  ...
```

Special case: typed vectors `vector<int>` and `vector<long>` also use the `0x1cb5c415` prefix. They do NOT use separate `int_vector` or `long_vector` (although the spec allowed it in early versions; layer 214 no longer does).

**Bare vector** (without a prefix): only allowed when a parent constructor declares it as `Vector<bare>`. The wire is then just `count + items` without `0x1cb5c415`. In practice rare; the codegen recognizes it by the `%(Vector ...)` syntax in the definition.

## 6. Pre-allocation and arenas

### Estimate-then-grow

Each generated struct exposes `EstimateSize()`, which returns an upper bound of the wire size. The codegen emits it by inductive construction:

```cpp
size_t TlUser::EstimateSize() const {
    size_t s = 4 + 4 + 8;       // ctor_id + flags + id
    if (flags & (1u << 0)) s += 8;
    if (flags & (1u << 1)) s += 4 + first_name.size() * 4 + 3; // worst case UTF-16 → UTF-8 + padding
    // ... rest ...
    return s;
}
```

Before serializing an RPC, application code calls:
```cpp
size_t estimated = method.EstimateSize();
TlWriter w(arena, estimated * 2);   // 2x factor for safety; the arena absorbs it
method.Write(w);
```

If `Write` exceeds the initial capacity, `TlWriter::EnsureCapacity` reallocates to the arena (not to the general heap). Arena reset at the end of the use case → zero leaks.

### Arena per request

```cpp
// In the MTProto send_rpc_use_case:
ArenaAllocator arena(/*page=*/8 KB);
TlWriter w(&arena, 4 KB);
auto rc = method.Write(w);
auto bytes = w.Bytes();
m_channel->SendEncrypted(bytes);
arena.Reset();              // frees everything
```

**Arena budget per RPC**: typically 4-16 KB. Large RPCs (`messages.sendMultiMedia` with 10 photos): ~100 KB. Arena page size configurable per context.

## 7. Error handling

`TlError` (in `domain/errors/tl_errors.h`):
```cpp
struct TlError {
    enum class Code : int32_t {
        Ok = 0,
        ReadOverflow = 2001,           // attempt to read past the end of the input
        WriteFailed = 2002,            // alloc failed
        WrongConstructor = 2003,       // unexpected constructor_id
        UnknownConstructor = 2004,     // ctor_id not in TlSchema
        UnsupportedType = 2005,        // type not implemented
        VectorMagicMismatch = 2006,    // 0x1cb5c415 missing
        StringTooLong = 2007,          // > 16 MB sanity cap
        BadAlignment = 2008,           // padding violation (defensive)
        FlagsBitsetInvalid = 2009,     // a bit set for an undeclared field
        UnexpectedEnd = 2010,
        SchemaMismatch = 2011,         // the layer does not support this ctor
    };
    Code code;
    int32_t aux;
    char detail[64];
};
```

Mapping to HRESULT at the boundary:
- `2001-2099` → `MAKE_HRESULT(SEVERITY_ERROR, FACILITY_ITF, 0x2000 + (code - 2000))`.

## 9. Public API surface (WinMD)

```cpp
namespace Vianium::Tl::v1 {

public ref class TlSchema sealed {
public:
    static TlSchema^ LoadLayer(int32_t layerNumber);
    property int32_t Layer { int32_t get(); }
    property int32_t ConstructorCount { int32_t get(); }
    bool HasConstructor(uint32_t ctorId);
    Platform::String^ ConstructorName(uint32_t ctorId);   // DEBUG only; release returns ""
};

public ref class TlSerializer sealed {
public:
    static TlSerializer^ Create(TlSchema^ schema);

    // The caller passes a ctor_id + a dictionary of fields. Native code looks it up in the schema and emits an IBuffer.
    // For the Vianigram MVP, serialization goes through a more efficient typed path (no reflection
    // dictionary). This API exists only for generic cases (e.g., debugging).
    Windows::Storage::Streams::IBuffer^ SerializeRaw(uint32_t ctorId,
        Windows::Foundation::Collections::IMap<Platform::String^, Platform::Object^>^ fields);

    Windows::Foundation::Collections::IMap<Platform::String^, Platform::Object^>^
        DeserializeRaw(Windows::Storage::Streams::IBuffer^ payload);
};

}
```

**Critical note:** the WinRT API is a **generic fallback**. The real usage path is C++ to C++ between the `src\tl\` and `src\mtproto\` subcomponents of the `vianium-mtproto` sibling via `vianium/tl/api/v1/tl_api.h` (C ABI). When `MTProtoChannel::SendAsync` receives an `IBuffer^ tlPayload` from managed, that buffer already comes serialized by **managed codegen** (a parallel set of C# helpers in `Vianium.Tl.Managed` that mirror the TL emit). That is to say:

- **Hot path:** managed C# code builds the `IBuffer` with TL serialized by C# code generated by `tl-codegen` (a second emitter, `--csharp` flag).
- **Validation:** the native `TlSchema` deserializes to validate the shape + dispatch the rpc_result → managed.
- **The codegen is bilingual:** a C# emit for serializing (a direct bridge to IBuffer); a C++ emit for deserializing and validating.

This duality is critical because building the TL in native would require crossing the boundary many times (one field at a time). The bilingual codegen eliminates that cost.

## 10. Self-tests (DEBUG)

At the first `TlSchema::LoadLayer()`:

1. **PrimitiveRoundtrip**: encode/decode of int, long, double, int128, int256 with known values. Byte-equal.
2. **StringRoundtripShort**: a string < 254 bytes. Verifies correct padding.
3. **StringRoundtripLong**: a string > 254 bytes. Verifies the 0xFE prefix + 3-byte length.
4. **VectorRoundtripInt**: a `vector<int>` with 10 elements. Verifies the 0x1cb5c415 magic.
5. **VectorRoundtripBoxed**: a `vector<User>` with 3 distinct users. Verifies that each item includes its ctor_id.
6. **FlagsRoundtrip**: an encoded message with some flag bits set, others not. Verifies that the unset ones do NOT appear in the wire.
7. **CtorIdHashCheck**: for 100 random constructors of the layer, validates CRC32(definition_text) == declared_ctor_id.
8. **BoundsChecks**: feeds a truncated input to a `TlReader`; verifies that each Read* returns `ReadOverflow` without crashing.
9. **NestingDepthCap**: a maliciously recursive input; verifies the cap at depth 32.
10. **UnknownConstructor**: creates a schema without a certain ctor_id; deserializing an input with that ctor → an UnknownConstructor error.

If it fails → `TlSchema::LoadLayer()` returns null + the counter `tl.selftest.failed_total{test=...}`.

## 11. Layer upgrade path

A layer bump (e.g., 214 → 220) implies:

1. Update `tools/tl-codegen/tl/scheme-layer-220.tl` (download from Telegram).
2. Re-run the codegen: emits `tl_layer_220.{h,cpp}`.
3. Update the sibling `vianium-mtproto\Vianium.Mtproto.vcxproj` to include the new .cpp.
4. Update the `TlSchema::LoadLayer(220)` factory.
5. **Breaking changes audited**: the codegen detects constructor_ids that changed between layers (same name, different hash). Reports a warning.
6. The managed layer is updated to build the new shapes.
7. Schema test: roundtrip of the top-100 RPCs of layer 220.

**Coexistence:** two layers do not coexist in the same `MTProtoChannel`. The channel is reopened with the new layer at the handshake (the `init_connection#c1cd5ea9` parameter carries `layer:int`).

**Policy for when to bump:**
- Telegram sometimes goes 6+ months without breaking layer bumps; the app then stays on layer N.
- If a required feature appears in layer N+M (e.g., a new media type), the bump is justified.
- An annual baseline bump to maintain compatibility.

## 14. Risks and mitigations

| Risk | Mitigation |
|--------|------------|
| The schema changes mid-conversation (the server bumps the layer without the client knowing) | The `tl.unknown_ctor_total` counter with alerting; the managed layer shows "update Vianigram" if it exceeds a threshold. |
| Historical constructor_id collision (rare but documented) | The codegen detects and rejects it if two distinct definitions share a ctor_id. |
| Maliciously crafted TL (overflow nesting, huge strings) | `domain/policies/string_length_policy` caps at 16 MB, `nesting_depth_policy` caps at 32. |
| A big TL message on the heap fragments | Arena per request → reset frees everything at once. |
| Bilingual codegen drift (C++ and C# emit differently) | CI test: roundtrip C# encode → C++ decode → C++ encode → byte-equal. |
| Build-time TL schema parsing fails due to a Telegram syntax change | The codegen tolerates documented variants; emits a warning + skip if it encounters an unknown construct. |

## 15. Phasing

**Phase 1 (ROADMAP):**
- Complete C# codegen tool.
- Lexer/parser of `.tl`.
- `TlReader`, `TlWriter` with bounds-checking.
- Generated code for layer 214 (~800 types, ~600 RPCs).
- Bilingual emitter (C++ and C# managed).
- Self-tests at DEBUG startup.
- Smoke test: roundtrip of the top-100 types.

**Phase 2:**
- Optimization: a pool of TlReader/TlWriter to avoid recreating the arena per call.
- Streaming reader for very large TL files (e.g., `messages.getHistory` with 100+ messages).

**Phase 4+:**
