# Vianigram.Core.Media — Opus, WebP, Image Scaling, Video Thumbnails

**Status:** planned (Phase 3) · **Last reviewed:** 2026-04-28

> Required prior reading:
> - [architecture-principles.md](architecture-principles.md) — toolchain, DDD+hex, WinMD ABI rules, memory budgets.
> - [00-overview.md](00-overview.md) — context map.
> - `..\..\THIRD_PARTY_NOTICES.md` — attributions for vendored code (libopus, libwebp).

`Vianigram.Core.Media` decodes and processes the multimedia formats that Telegram uses: voice (Opus), images (WebP, JPEG), video thumbnails, and animated stickers (TGS Lottie, deferred). It is pure CPU: no network, no storage. All of its inputs are `IBuffer` with bytes, all of its outputs are `IBuffer` with decoded or re-encoded data.

---

## 1. Bounded context

### Ubiquitous language

| Term | Definition |
|---------|------------|
| **opus_frame** | A 20 ms Opus frame (= 320 samples at 16 kHz mono or 960 samples at 48 kHz). |
| **opus_packet** | A concatenation of one or more frames with a TOC byte. What goes over the network in voice notes. |
| **webp_image** | A WebP image — VP8 lossy, VP8L lossless, or VP8X extended. Telegram uses all three. |
| **sticker** | A WebP image (static) or TGS (animated Lottie). |
| **thumbnail** | A reduced image, typically ≤ 320×320 px, generated by the client. |
| **video_keyframe** | An I-frame of an MP4 H.264 video; used for the thumb instead of a full decode. |
| **TGS** | Telegram Animated Sticker — Lottie JSON compressed with gzip, extension .tgs. |
| **stripped_size** | Photo size ID `i` (single-byte stripped): a WebP-compressed thumbnail < 64 bytes that the server delivers inline. The decoder reconstructs it. |

### Aggregate root

`MediaDecoder` (in `domain/aggregates/media_decoder.h`):
- Stateless from the point of view of the API; each `Decode*` is self-contained.
- Owns internally: static tables of Opus codebooks, WebP Huffman tables, a bounded LRU cache of decoded thumbnails.
- Invariants:
  - Total memory cap ≤ 24 MB hot.
  - Self-tests at startup; refuses service if they fail.
  - The LRU cache evicts the oldest when it exceeds the budget.

### Domain events

- `DecodeStarted{ format, input_bytes }`.
- `DecodeCompleted{ format, output_bytes, duration_ms }`.
- `DecodeFailed{ format, reason }`.
- `ThumbnailReady{ source_kind, width, height, bytes }`.
- `CacheEvicted{ key, age_ms }`.

## 2. Opus decode

### Spec

RFC 6716 (Opus codec). Telegram uses Opus for:
- **Voice notes**: 16 kHz mono, ~16-32 kbps. 20 ms frames. Single-stream.
- **Voice calls** (out of scope for this context — that is VoIP, see `05-voip.md`): 48 kHz mono, 16-32 kbps adaptive.

Vianigram decodes only voice notes in this context. VoIP shares the base codec (the same decoder) but lives in `Vianium.VoIP` with per-session state.

### Vendoring strategy

**libopus** (BSD 3-clause) is the reference implementation. Textual vendoring approved per `THIRD_PARTY_NOTICES.md`:

```
Core/Vianigram.Core.Media/src/infrastructure/codecs/opus/
├── celt/                      ← CELT layer (high-quality)
├── silk/                      ← SILK layer (low-bitrate speech)
├── opus/                      ← top-level Opus
└── (selective subset; the full version is not compiled to reduce footprint)
```

**Compiled subset** (decoder-only path):
- `silk_decoder.{h,c}` and dependencies.
- `celt_decoder.{h,c}` and dependencies.
- `opus_decoder.{h,c}` (wrapper).
- Embedded codebook tables.
- **NOT**: encoder paths, repacketizer, multistream (out of scope for voice notes).

**Memory budget** (estimate):
- Opus decoder state: ~24 KB per instance.
- Codebooks: ~70 KB shared (read-only static tables).
- Working frame buffer: ~4 KB.
- Total per decode: ~100 KB scratch + shared static.

**Implementation wrapper:**
```
src/infrastructure/codecs/opus_decoder_adapter.{h,cpp}
```

```cpp
namespace vianigram::media::infrastructure {

class OpusDecoderAdapter {
public:
    OpusDecoderAdapter(IAllocator* alloc, int sample_rate = 16000, int channels = 1);
    ~OpusDecoderAdapter();

    // Decode complete voice note (concatenated Opus packets).
    Result<Span<int16_t>, MediaError> DecodeAll(
        Span<const uint8_t> opus_data,
        IAllocator* outAlloc);

    // Streaming: decode one packet at a time.
    Result<size_t /* samples_written */, MediaError> DecodePacket(
        Span<const uint8_t> packet,
        Span<int16_t> out_pcm);

private:
    OpusDecoder* m_decoder;          // libopus state
    IAllocator* m_alloc;
    int m_sampleRate;
    int m_channels;
};

}
```

### Self-test

RFC 6716 Annex A.5 reference test vectors. 12 test vectors total. Decoder roundtrip → verify PCM matches reference byte-equal (or within tolerance for floating-point modes; the INT16 mode used here is byte-equal).

## 3. WebP decode

### Spec

- **VP8 lossy**: a RIFF container with a 'VP8 ' chunk. Used for stickers and photos.
- **VP8L lossless**: a 'VP8L' chunk. Used for high-quality stickers and small images.
- **VP8X extended**: a 'VP8X' chunk with possible alpha, animation, EXIF. Telegram uses it for animated WebP (stickers).

### Vendoring strategy

**libwebp** (BSD 3-clause) is the reference implementation. Vendoring approved.

```
Core/Vianigram.Core.Media/src/infrastructure/codecs/libwebp/
├── decode/                    ← decoder paths only
├── dsp/                       ← optimized primitives (NEON for ARM)
├── utils/                     ← bit readers, color conversion
└── (encoder NOT compiled — Vianigram only decodes WebP, no encode)
```

**Memory budget:**
- Decoder state: ~50 KB.
- Working buffers (Y/U/V planes for VP8, RGB for output): proportional to image size. For a typical 512×512 sticker: 768 KB temp + 1 MB output RGBA.
- Huffman tables (VP8L): ~100 KB.

**Implementation wrapper:**
```
src/infrastructure/codecs/webp_decoder_adapter.{h,cpp}
```

```cpp
class WebpDecoderAdapter {
public:
    struct DecodedImage {
        int width;
        int height;
        bool has_alpha;
        Span<uint8_t> rgba_bytes;       // width * height * 4 bytes
    };

    static Result<DecodedImage, MediaError> Decode(
        Span<const uint8_t> webp_bytes,
        IAllocator* outAlloc);

    // Streaming for large images (animated WebP frames).
    static Result<DecodedImage, MediaError> DecodeFrame(
        Span<const uint8_t> webp_bytes,
        int frame_index,
        IAllocator* outAlloc);

    static Result<int /* frame_count */, MediaError> ProbeFrameCount(
        Span<const uint8_t> webp_bytes);
};
```

## 4. JPEG decode (auxiliary)

Telegram also sends photos as JPEG (most photos > 100 KB are JPEG). Vianigram delegates JPEG decoding to the system (`Windows.Graphics.Imaging.BitmapDecoder`) to avoid vendoring libjpeg-turbo. Reasons:

- libjpeg-turbo is robust but heavy (~500 KB code).
- The Windows BitmapDecoder is hardware-accelerated on some chipsets.
- JPEG is not performance-critical (photos are loaded async with a UI placeholder).

**Implementation:**
```
src/infrastructure/codecs/jpeg_decoder_winrt.{h,cpp}
```

```cpp
class JpegDecoderWinRt {
public:
    static Windows::Foundation::IAsyncOperation<DecodedImage^>^ DecodeAsync(
        Windows::Storage::Streams::IBuffer^ jpeg_bytes);
};
```

If, in the future, the system introduces a regression (rare but possible in updates), Vianigram can substitute a vendored libjpeg-turbo without changing the caller.

## 5. Image scaling (bilinear interpolation)

### Use cases

- Generate thumbnails of full-size photos for chat list previews (full size → 320×320).
- Generate previews for sent media (full → multiple sizes per server requirements).
- Resize WebP stickers for different display densities (a 512×512 sticker → 256×256 if the UI requests it).

### Algorithm

Bilinear interpolation, SIMD-optimized:
- ARM target: NEON intrinsics (`<arm_neon.h>`).
- x86 target (emulator/sim): SSE2 intrinsics (`<emmintrin.h>`).
- Scalar fallback path for debug builds.

**Implementation:**
```
src/domain/services/image_scaler.{h,cpp}
src/infrastructure/codecs/image_scaler_neon.cpp        ← compiled only for ARM
src/infrastructure/codecs/image_scaler_sse2.cpp        ← compiled only for x86
src/infrastructure/codecs/image_scaler_scalar.cpp      ← fallback
```

```cpp
class ImageScaler {
public:
    enum class PixelFormat {
        Rgba8888,
        Bgra8888,
        Rgb888,
    };

    static Result<Span<uint8_t>, MediaError> ScaleBilinear(
        Span<const uint8_t> src, int src_w, int src_h, PixelFormat fmt,
        int dst_w, int dst_h,
        IAllocator* outAlloc);
};
```

**Memory budget:** a workspace of ~2x the dst size during the operation; arena reset at the end.

**Performance target** (Snapdragon S4):
- 1024×1024 RGBA → 320×320: < 30 ms NEON, < 80 ms scalar.
- 320×320 → 64×64 (chat list): < 5 ms NEON.

### Self-test

Reference 4×4 RGBA → 2×2 (a trivial bilinear known result), then 256×256 → 128×128 with a synthetic gradient (verify smooth transitions, max delta < 2 per channel).

## 6. Video thumbnail extraction

### Approach

Vianigram does NOT decode the full video. For a thumbnail of an MP4 H.264, a **minimal** parser is used that:
1. Finds the `mdat` box.
2. Locates the first keyframe (I-frame) near t=0.5s.
3. Extracts the raw bytes of the NAL units of that frame.
4. **Does not decode the H.264**. Instead, the Telegram client apps typically generate the thumb server-side and send it together with the video metadata. If the client needs to re-generate it:
   - Option A: use `Windows.Media.Editing.MediaComposition` to extract a keyframe (decodes with hardware if available).
   - Option B: a minimal parser that extracts bytes but does NOT decode → only useful if the MP4 includes a separate thumbnail track.

The Vianigram MVP uses **Option A** (the Windows API), which is already available on WP8.1:

```
src/infrastructure/codecs/video_thumb_winrt.{h,cpp}
```

```cpp
class VideoThumbWinRt {
public:
    static Windows::Foundation::IAsyncOperation<DecodedImage^>^ ExtractFirstKeyframeAsync(
        Windows::Storage::Streams::IBuffer^ mp4_bytes,
        Windows::Foundation::TimeSpan position);   // typically TimeSpan{5000000} = 0.5s
};
```

Internally uses `Windows.Media.Editing.MediaComposition::CreateFromUriAsync` + `GetThumbnailAsync`. The output is a JPEG `IBuffer` that Vianigram decodes with `JpegDecoderWinRt`.

**Phase 3 scope:** Option A only. If telemetry shows that the server always provides a pre-encoded thumb (likely), this path is redundant. **Phase 4 decision** based on data.

### Phase 5+ deferred: H.264 minimal decoder

If telemetry reveals cases where the server does NOT deliver a thumb and the Windows API fails, vendor a minimal H.264 decoder (just I-frames, no P/B). Candidates: minihevcvld for HEVC, a custom port of OpenH264 (BSD-2). Out of scope until justified.

## 7. TGS animated stickers

### Format

`.tgs` = Lottie JSON (Adobe Lottie format) compressed with gzip. Lottie describes vector animations by frames.

### Phase 5+ deferred

Phase 3 implements a **placeholder**: Vianigram detects TGS by extension/MIME and shows the first frame as a static WebP fallback (Telegram sends a WebP thumbnail along with the TGS).

**Implementation Phase 3:**

- Render to a bitmap at the display rate (typically 30 fps).

## 8. Bounded LRU cache

```
src/domain/services/lru_cache.{h,cpp}
```

```cpp
template <typename Key, typename Value>
class LruCache {
public:
    LruCache(IAllocator* alloc, size_t max_bytes);
    void Put(Key key, Value value, size_t value_bytes);
    Maybe<Value> Get(Key key);
    void Clear();
    size_t BytesUsed() const;
};
```

**Cache layers:**
- `m_thumbnailCache`: reconstructed stripped-size → decoded RGBA. Cap 16 MB.
- `m_decodedStickerCache`: WebP sticker file_id → decoded RGBA. Cap 4 MB (a small set of recently-viewed stickers).
- Decoder state caches are NOT done (decoders are cheap to recreate; the I/O is the cost).

Cache key derivation: `MediaCacheKey = SHA-256(file_id_64 || access_hash_64)[0..15]`. Avoids storing access hashes in the cache key.

**Eviction policy:** strict LRU. On a cache hit, move to front. On Put, if `BytesUsed + value_bytes > max_bytes`, evict from the back until it fits.

## 11. Self-tests (DEBUG)

At the first `MediaDecoder::Create`:

1. **OpusRfc6716TestVectors**: 12 RFC 6716 Annex A.5 reference packets → decode → verify INT16 PCM byte-equal.
2. **WebpVp8DecodeReference**: Google reference VP8 lossy → verify RGBA output.
3. **WebpVp8lDecodeReference**: VP8L lossless → verify RGBA output.
4. **WebpVp8xAlphaDecode**: VP8X with an alpha channel → verify the alpha channel.
5. **JpegBaselineDecode**: an SOF0 baseline JPEG → verify dimensions + first row checksum.
6. **StrippedThumbReconstruct**: known stripped bytes → reconstruct → decode → verify dimensions.
7. **ImageScalerBilinear4x4to2x2**: a trivial known-result test.
8. **ImageScalerNeonVsScalar**: a random 256x256 image, scale to 128x128 with both the NEON and scalar paths, verify max per-channel delta ≤ 1 (rounding).
9. **LruCacheBasic**: insert N items, exceed the budget, verify the oldest is evicted.
10. **MaxImageDimensionsPolicy**: feed an 8192x8192 image → verify it is rejected by the policy.

If any of them fails → `MediaDecoder::Create` returns null + `media.selftest.failed_total{test=...}` increments. The managed layer shows a degraded-mode banner.

## 13. Risks and mitigations

| Risk | Mitigation |
|--------|------------|
| libopus / libwebp have an RCE vulnerability on malicious input | (a) `max_image_dimensions_policy` caps at 4096x4096; (b) `decoder_timeout_policy` cancels after 2s; (c) updates when a CVE appears — vendored but with the commit hash tracked in `THIRD_PARTY_NOTICES.md`. |
| The LRU cache runs unbounded due to a bug | `BytesUsed()` gauge + alerting if > 1.5x the cap. Defense in depth: `Put` checks before insert. |
| The image scaler NEON paths produce a slightly different output vs scalar | Self-test with a tolerance of 1 per channel; CI verifies. If the diff exceeds the tolerance, the build fails. |
| The WP8.1 BitmapDecoder breaks on a certain device → JPEG decode fails | A fallback path that vendors libjpeg-turbo. Telemetry detects the failure rate. |
| The TGS Lottie spec changes between versions (Telegram bumps the format) | The Phase 3 placeholder side-steps the issue (always uses the WebP fallback). Phase 5+ must track spec changes. |
| Memory fragmentation from large allocations (1 MB RGBA per sticker) | Arena per decode → reset at the end. The cache uses the heap with a bounded LRU. |
| The stripped-thumb pre-shared header is out of date due to a server change | The canonical Telegram spec published at https://core.telegram.org/api/files; counter `media.stripped_thumb.reconstruction_failures` for alerting. |
| Video thumb extraction via `MediaComposition` consumes a lot of RAM on large videos | Cap input size at 50 MB; videos > 50 MB are not thumbnailed by the client, the server-provided thumb is trusted. |

## 14. Phasing

**Phase 3 (ROADMAP):**
- libopus subset vendored + adapter.
- libwebp subset vendored + adapter.
- JPEG decoder via WinRT.
- Stripped thumb reconstructor.
- Image scaler with scalar + NEON + SSE2 paths.
- LRU cache for thumbnails and stickers.
- Self-tests at DEBUG startup.
- Smoke test: download a photo from the CDN → decode → display in chat.
- Complete WinMD shim.

**Phase 4:**
- Video thumb extraction via WinRT.
- TGS placeholder (static WebP fallback).

**Phase 5+:**
- Full TGS Lottie renderer (rlottie vendored or custom).
- H.264 keyframe decoder if telemetry justifies it.
- Animated WebP playback (multiple frames).

**Deferred / out of scope:**
- Opus encoder (Vianigram only decodes voice notes; voice send goes through `Vianium.VoIP`, which has its own encoder for calls, NO encoding for voice notes — those are recorded via `Windows.Media.Capture`, which produces a stream that the client can re-encode as Opus via `Windows.Media.MediaProperties`).
- WebP encoder (Vianigram never sends custom WebP — stickers/thumbs come from the server).
- Image format conversion utilities (the managed side is fine).
- HDR / 10-bit color (not relevant to Telegram WP8.1).
